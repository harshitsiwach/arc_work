## 5. DATABASE SCHEMA (append to supabase_setup.sql — DO NOT replace existing tables)
================================================================================

-- Bounties table
CREATE TABLE bounties (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  creator_id UUID REFERENCES auth.users(id) ON DELETE CASCADE,
  title TEXT NOT NULL,
  description TEXT NOT NULL,
  reward_usdc NUMERIC(18,6) NOT NULL,
  deadline TIMESTAMPTZ NOT NULL,
  worker_type TEXT NOT NULL CHECK (worker_type IN ('HUMAN', 'AGENT', 'BOTH')),
  status TEXT NOT NULL DEFAULT 'OPEN' CHECK (status IN ('OPEN', 'FUNDED', 'SUBMITTED', 'COMPLETED', 'REFUNDED', 'DISPUTED')),
  winner_id UUID REFERENCES auth.users(id),
  contract_bounty_id TEXT,
  escrow_tx_hash TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- Submissions table
CREATE TABLE bounty_submissions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  bounty_id UUID REFERENCES bounties(id) ON DELETE CASCADE,
  submitter_id UUID REFERENCES auth.users(id) ON DELETE CASCADE,
  proof_hash TEXT NOT NULL,
  proof_url TEXT,
  notes TEXT,
  status TEXT NOT NULL DEFAULT 'PENDING' CHECK (status IN ('PENDING', 'APPROVED', 'REJECTED')),
  ai_validation_score NUMERIC,
  ai_validation_notes TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- RLS
ALTER TABLE bounties ENABLE ROW LEVEL SECURITY;
ALTER TABLE bounty_submissions ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Anyone can read open bounties" ON bounties FOR SELECT USING (true);
CREATE POLICY "Creator can insert bounty" ON bounties FOR INSERT WITH CHECK (auth.uid() = creator_id);
CREATE POLICY "Creator can update own bounty" ON bounties FOR UPDATE USING (auth.uid() = creator_id);
CREATE POLICY "Anyone can read submissions" ON bounty_submissions FOR SELECT USING (true);
CREATE POLICY "Submitter can insert" ON bounty_submissions FOR INSERT WITH CHECK (auth.uid() = submitter_id);

## 6. SMART CONTRACT (contracts/BountyEscrow.sol)
================================================================================

Write NEW file — do NOT modify existing contracts in /contract.
Use Solidity ^0.8.20 with OpenZeppelin.

Libraries: SafeERC20, ReentrancyGuard, AccessControl, Pausable

Roles: MODERATOR_ROLE, ADMIN_ROLE

Enums:
  WorkerType  { HUMAN, AGENT, BOTH }
  BountyStatus { OPEN, FUNDED, SUBMITTED, COMPLETED, REFUNDED, DISPUTED }

Structs:
  Bounty {
    uint256 id;
    address creator;
    uint256 reward;
    uint256 deadline;
    WorkerType workerType;
    BountyStatus status;
    address winner;
    uint256 submissionCount;
  }

  Submission {
    uint256 id;
    uint256 bountyId;
    address submitter;
    bytes32 proofHash;
    uint256 submittedAt;
    bool isApproved;
  }

Functions:
  createBounty(string description, uint256 reward, uint256 deadline, WorkerType workerType)
    → transfers USDC from creator into contract escrow, emits BountyCreated, returns bountyId

  submitWork(uint256 bountyId, bytes32 proofHash)
    → stores submission, increments submissionCount, emits WorkSubmitted

  approveSubmission(uint256 bountyId, uint256 submissionId)
    → MODERATOR_ROLE only, releases USDC to submitter, sets COMPLETED, emits SubmissionApproved + BountyCompleted

  rejectSubmission(uint256 bountyId, uint256 submissionId)
    → MODERATOR_ROLE only, emits SubmissionRejected

  refundCreator(uint256 bountyId)
    → callable by anyone after deadline if status != COMPLETED, returns USDC to creator, emits BountyRefunded

  raiseDispute(uint256 bountyId)
    → callable by creator or submitter, sets DISPUTED, emits DisputeRaised

  getBounty(uint256 id) view returns (Bounty memory)
  getSubmissions(uint256 bountyId) view returns (Submission[] memory)

Security: ReentrancyGuard on all state-changing functions. SafeERC20 for all transfers.

Events: BountyCreated, WorkSubmitted, SubmissionApproved, SubmissionRejected,
        BountyCompleted, BountyRefunded, DisputeRaised

## 8. HOOKS (hooks/useBounty.ts) — NEW FILE
================================================================================

Reuse existing viem adapter pattern from lib/web3/appkit-provider.tsx.
DO NOT create a new wallet provider. Import contract ABI from constants/BountyEscrowABI.ts.
Import Supabase client from existing lib/supabase (or wherever it already lives).
Import OpenAI client from existing setup.

--- useCreateBounty() ---
  1. Call USDC.approve(bountyEscrowAddress, reward) via existing wallet
  2. Call BountyEscrow.createBounty(description, reward, deadline, workerType)
  3. Insert row into Supabase `bounties` table with returned contract bounty ID + tx hash
  Returns: { createBounty(params), isLoading, error, txHash }

  params type:
  {
    title: string
    description: string
    rewardUsdc: string        // human-readable, e.g. "50.00"
    deadline: Date
    workerType: 'HUMAN' | 'AGENT' | 'BOTH'
  }

--- useSubmitWork() ---
  1. Accept File object from user
  2. Upload to IPFS via https://api.web3.storage (token: NEXT_PUBLIC_WEB3_STORAGE_TOKEN)
  3. Convert returned CID to bytes32
  4. Call BountyEscrow.submitWork(bountyId, proofHash)
  5. Insert row into Supabase `bounty_submissions`
  6. POST to /api/bounties/validate with { proofUrl, bountyDescription, submissionId }
     (fire-and-forget, don't block the tx)
  Returns: { submitWork(bountyId, file, notes?), isLoading, error, txHash }

--- useBounties(filters?) ---
  filters type: { status?: string, workerType?: string, minReward?: number, maxReward?: number }
  1. Query Supabase `bounties` table with filters
  2. Subscribe to Supabase Realtime on `bounties` table for live inserts/updates
  3. Unsubscribe on unmount
  Returns: { bounties, isLoading, error }

--- useBountyById(id: string) ---
  1. Fetch single bounty from Supabase by id
  2. Fetch all submissions for that bounty
  Returns: { bounty, submissions, isLoading, error }

--- useApproveSubmission() ---
  1. Call BountyEscrow.approveSubmission(bountyId, submissionId) (requires MODERATOR_ROLE wallet)
  2. Update Supabase: submission status → 'APPROVED', bounty status → 'COMPLETED', winner_id → submitter_id
  Returns: { approve(bountyId, submissionId), isLoading, error }

--- useRefundCreator() ---
  1. Call BountyEscrow.refundCreator(bountyId)
  2. Update Supabase: bounty status → 'REFUNDED'
  Returns: { refund(bountyId), isLoading, error }

## 9. API ROUTE (app/api/bounties/validate/route.ts)
================================================================================

POST handler only. Body: { proofUrl: string, bountyDescription: string, submissionId: string }

1. Call OpenAI (reuse existing client) with:
   - Model: gpt-4o (vision capable)
   - Prompt: "You are validating a bounty submission. Bounty description: [bountyDescription].
     The submitter provided this proof: [proofUrl].
     Rate how well this proof satisfies the bounty requirements.
     Return JSON: { score: number between 0 and 1, notes: string (1-2 sentences) }"
2. Parse response → { score, notes }
3. Update Supabase `bounty_submissions`:
   SET ai_validation_score = score, ai_validation_notes = notes WHERE id = submissionId
4. Return { score, notes }

## 14. COMPLETE BOUNTY LIFECYCLE (end-to-end flow)
================================================================================

POST A BOUNTY:
  1. User opens Marketplace → clicks Bounties tab
  2. Clicks "Post Bounty" → PostBountyModal opens
  3. Fills form → Step 2: approves USDC + calls BountyEscrow.createBounty()
  4. USDC locked in contract escrow
  5. Supabase row inserted with contract_bounty_id + escrow_tx_hash
  6. Bounty appears live on board (Supabase Realtime pushes to all viewers)

SUBMIT WORK:
  7. Another user (human or AI agent) clicks bounty → BountyDetailPage
  8. Clicks "Submit Work" → SubmissionModal
  9. Uploads file → goes to IPFS → CID converted to bytes32
  10. BountyEscrow.submitWork(bountyId, proofHash) called on-chain
  11. Supabase submission row inserted
  12. /api/bounties/validate fires → OpenAI scores the submission → score saved to DB
  13. Creator sees submission with AI score in BountyDetailPage

RESOLUTION:
  APPROVE:  Moderator clicks Approve → BountyEscrow.approveSubmission()
            → USDC released to winner's wallet → bounty COMPLETED
  REJECT:   Moderator rejects → submitter can resubmit (if deadline not passed)
  DISPUTE:  Creator or submitter raises dispute → 48hr manual review window
  REFUND:   Deadline passes, no winner → creator calls refundCreator() → USDC returned

## 15. AI AGENT PARTICIPATION (key differentiator vs Pump.fun GO)
================================================================================

Pump.fun GO only allows humans. Your app allows AI agents natively:

- Agents can POST bounties: createBounty() called from agent wallet
- Agents can SUBMIT to bounties: submitWork() called from agent contract address
  → contract checks the submitter address, doesn't care if it's human or smart contract
- Agent capability matching: when bounty workerType=AGENT, match against ERC-8004
  capability claims in .agents/skills to auto-suggest or auto-bid on bounties
- Agents get paid directly to their wallet/contract address via approveSubmission()
